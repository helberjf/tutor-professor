# Project Architecture

This document outlines the overall architecture of the Tutor and Professor project, detailing its components, their interactions, and the technologies used.

## Monorepo Structure

The project is organized as a monorepo, which means all related projects (frontend, backend, shared utilities) are stored in a single repository. This approach offers several benefits:

*   **Simplified Code Sharing**: Easy sharing of code, types, and configurations between applications.
*   **Atomic Commits**: Changes across different parts of the system can be made and committed together.
*   **Consistent Tooling**: A unified development environment and build process.
*   **Easier Refactoring**: Changes in shared code can be propagated and tested across all dependent projects more easily.

### Directory Layout

```
english-kids-tutor/
  apps/             # Contains independent applications
    web/            # Next.js Frontend
    api/            # FastAPI Backend
  packages/         # Reusable packages/libraries (e.g., shared types, UI components)
    shared/         # Currently empty, ready for common code
  content/          # Educational content (lessons, quizzes, stories)
  infra/            # Infrastructure configurations (Docker, Cloudflare)
  docs/             # Project documentation
  scripts/          # Automation scripts
  .gitignore
  README.md
  docker-compose.yml
```

## Frontend (`apps/web`)

The frontend is a **Next.js** application built with **React** and **TypeScript**. It utilizes the **App Router** for routing and **Tailwind CSS** for styling. The primary goal of the frontend is to provide a simple, intuitive, and engaging user interface for children and a straightforward management area for parents.

### Key Technologies:

*   **Next.js**: A React framework for building full-stack web applications. Chosen for its server-side rendering (SSR), static site generation (SSG), and API routes capabilities, which are beneficial for performance and SEO.
*   **React**: A JavaScript library for building user interfaces.
*   **TypeScript**: A superset of JavaScript that adds static typing, improving code quality and maintainability.
*   **Tailwind CSS**: A utility-first CSS framework for rapidly building custom designs. It helps in creating a visually appealing and responsive interface without writing custom CSS.
*   **HTML5 Audio**: For playing text-to-speech audio generated by the backend.

### Pages and Components:

The frontend includes pages for the home screen, daily lessons, review sessions, progress tracking, and a dedicated area for parents. Reusable components like `Card`, `Button`, `AudioPlayer`, and `ParentSettingsForm` ensure consistency and maintainability.

## Backend (`apps/api`)

The backend is a **FastAPI** application written in **Python 3.11+** with **Pydantic** for data validation. It serves as the central hub for business logic, data storage, and integration with external services like Kokoro TTS. The backend is designed to be modular and scalable, allowing for future expansion and changes in underlying technologies.

### Key Technologies:

*   **FastAPI**: A modern, fast (high-performance) web framework for building APIs with Python 3.7+ based on standard Python type hints. It offers automatic interactive API documentation (Swagger UI/ReDoc).
*   **Pydantic**: Used for data validation and settings management, ensuring data integrity for API requests and responses.
*   **SQLModel**: A library for interacting with SQL databases, combining the best of SQLAlchemy and Pydantic. It provides an elegant way to define database models and schemas.
*   **SQLite**: Used as the initial database for simplicity and ease of local development. The architecture is designed to allow for a smooth migration to PostgreSQL in the future.
*   **APScheduler**: A flexible in-process task scheduler for Python, used for scheduling daily lesson generation and other background tasks.
*   **Kokoro TTS**: A local text-to-speech engine integrated for generating audio from text. The backend provides an abstraction layer to allow easy swapping of TTS providers.

### Backend Structure:

```
apps/api/
  main.py           # FastAPI application entry point
  core/             # Core application logic, configurations
  api/routes/       # API endpoint definitions
  models/           # Database models (SQLModel)
  schemas/          # Pydantic schemas for request/response validation
  services/         # Business logic and external service integrations (e.g., TTS)
  db/               # Database connection and session management
  prompts/          # AI tutor system prompts
  utils/            # Utility functions
```

### API Routes:

The backend exposes a set of RESTful API endpoints for managing child profiles, lessons, reviews, progress, chat interactions, audio generation, and parent settings.

## Data Storage

*   **SQLite**: The initial choice for the database due to its file-based nature, making local setup straightforward. It stores child profiles, lesson data, review items, quiz attempts, progress snapshots, parent settings, and audio cache metadata.
*   **Future Migration**: The database models are structured to facilitate a future migration to a more robust relational database like **PostgreSQL** without significant code changes.

## Text-to-Speech (TTS)

*   **Kokoro TTS**: The primary TTS engine. It runs as a separate local service, and the FastAPI backend communicates with it to generate audio files. These files are then cached locally to reduce redundant generation.
*   **Modular Design**: The TTS integration is designed to be modular, allowing for easy replacement with other TTS providers if needed.

## Authentication

*   **Simple Parent Authentication**: A basic password-based authentication system for the parent area, using a password stored in environment variables or hashed in the database. It employs simple session management.
*   **Security**: Basic route protection is implemented for sensitive parent settings.

## Deployment and Infrastructure

*   **Vercel (Frontend)**: The Next.js frontend is designed for seamless deployment to Vercel, leveraging its continuous deployment features and global CDN.
*   **Local Backend / Cloudflare Tunnel**: The FastAPI backend is intended to run locally and be exposed to the internet via **Cloudflare Tunnel**. This provides a secure way to connect the Vercel-deployed frontend to a local backend without opening firewall ports.
*   **Docker**: Dockerfiles and a `docker-compose.yml` are provided for both the frontend and backend, enabling containerized development and deployment. This ensures consistent environments across different machines.

## Interaction Flow

1.  **User (Child)** interacts with the Next.js frontend in their browser.
2.  **Frontend** makes API calls to the FastAPI backend for data (lessons, progress) and actions (complete lesson, chat).
3.  **FastAPI Backend** processes requests, interacts with the SQLite database, and communicates with the local Kokoro TTS server for audio generation.
4.  **Kokoro TTS** generates audio files, which are cached by the backend and served to the frontend.
5.  **Parents** access a protected area in the frontend to manage child settings, authenticating against the backend.
6.  **Cloudflare Tunnel** (when configured) securely routes external requests from the Vercel frontend to the local FastAPI backend.

This architecture provides a robust, maintainable, and extensible foundation for the Tutor and Professor application.
